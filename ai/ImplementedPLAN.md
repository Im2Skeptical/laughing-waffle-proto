# Plan: Expand `devPlaytesting01` into a hidden-frontier onboarding sequence

## Summary
Implement `devPlaytesting01` as a staged discovery flow driven by deterministic work processes and explicit discovery state, without changing the authoritative board size or replay model.

The onboarding path will be:
1. Start with only env col `0` visible, shown as `???` with `Explore`.
2. Completing `Explore` reveals that tile’s true title/tags, grants a deterministic sample package, and exposes adjacent hidden env cols as new `??? + Explore` frontier tiles.
3. In this scenario, exposing env col `1` also reveals the board envStructure `Ancient Ruins`.
4. Exploring env col `1` reveals `Levee`, grants its sample package, exposes env col `2`, and reveals `Delve` on the Levee tile.
5. Completing `Delve` reveals the hub row and the hidden hub structure `Temple Ruins`.
6. `Temple Ruins` already has an active zero-material `build` process and a visible `build` tag; occupying it advances a 5-second rebuild into `Makeshift Shelter`.
7. When that rebuild completes, the structure transforms in place, keeps its inventory, unlocks hub renaming, and the shared hub/ruins area name auto-changes from `Ancient Ruins` to `Hub`.

## Impact Analysis
- Determinism: sample rewards must be generated by repeated `state.rng` rolls at process completion time; no `Math.random()`.
- Serialization: add only plain JSON state for discovery/visibility/tag hidden-state and process completion payloads.
- Replay: all reveal/unlock/build transitions must happen through authoritative process completion in simulation ticks so `rebuildStateAtSecond(tSec)` reproduces them exactly.
- Layering: discovery rules live in model/state/effects; views only read visibility/title state and gate rendering/input.

## Public API / Interface / Type Changes
- Add a JSON-serializable discovery slice to `GameState`, for example:
  - `state.discovery.envCols[col] = { exposed: boolean, revealed: boolean }`
  - `state.discovery.hubVisible: boolean`
  - `state.discovery.hubRenameUnlocked: boolean`
- Extend local tag state to support hidden-but-present tags, parallel to disabled tags:
  - `target.tagStates[tagId].hiddenBy.discovery === true`
- Extend work processes with a generalized completion payload:
  - `process.onComplete` or `process.completionEffects`
- Add generalized effect ops needed by that completion payload:
  - reveal/expose discovery state
  - hide/reveal tags
  - set location name
  - spawn deterministic sample package from a drop table spec
- Extend scenario instance specs in `scenarios-defs` / `init.js` so tile/envStructure/hubStructure setup objects can override serializable instance fields such as `tags`, `tagStates`, `props`, and `systemState`.
- Extend view title resolvers so envStructure UI title functions can read `state`, allowing the Levee ruins marker to mirror the current hub area name.

## Model / Content Work
### 1. Discovery state and hidden-tag semantics
- Keep `state.board.cols = 12` authoritative.
- Add discovery state for env columns and hub visibility instead of mutating board size.
- Hidden env columns are not rendered and are not interactable.
- Exposed-but-unrevealed tiles render as generic `???` cards.
- Revealed tiles render normal def title/color/description.
- Tag execution and tag UI must ignore locally hidden tags the same way they already ignore disabled tags.

### 2. Generic process-completion effects
- Add a generalized process completion hook so timed work can execute arbitrary effect lists on completion.
- Use that hook for `Explore` and `Delve`; do not hardcode scenario-specific completion branches in execution code.

### 3. New env tags
- Add `explore` env tag.
- Add `delve` env tag.
- Each uses a single auto intent that, every second while occupied:
  - pays `1` stamina
  - `CreateWorkProcess` with `uniqueType: true`
  - `AdvanceWorkProcess` in the same intent
- Duration:
  - `Explore`: `5` seconds
  - `Delve`: `5` seconds
- On completion:
  - `Explore` removes itself from that tile, reveals the tile, unhides that tile’s other tags, grants the sample package, and exposes adjacent hidden env cols as frontier.
  - `Delve` removes itself and sets `hubVisible = true`.

### 4. Sample package rewards
- Add a reusable drop-package effect that defaults to repeated forage rolls.
- Default rule: `15` deterministic rolls against that tile’s forage table.
- Allow per-tile override for future content, but use the default in this scenario unless a tile explicitly overrides it.
- Target the acting pawn inventory.

### 5. Scenario content changes for `devPlaytesting01`
- Move the pawn start from hub row to `envCol: 0`.
- Remove direct spawn of `moteOfEternity` and `mysteriousAncientTome` into pawn inventory.
- Replace the starting hub structure with a hidden `templeRuins` hub structure at `hubCol: 4`.
- Put the tome + mote into the `Temple Ruins` inventory.
- Replace `hubPortal` usage in this scenario with a dedicated board envStructure def such as `ancientRuins` at env col `1`.
- Set initial shared hub area name to `Ancient Ruins`.
- Hide the hub row initially.
- Seed all scenario env tiles for frontier exploration:
  - all tiles have `explore` in tag order ahead of hidden base tags
  - only col `0` starts `exposed: true`
  - only explored cols become `revealed: true`
- On env col `1`, include `delve` as an additional hidden local tag, ordered before `forageable`, so when the Levee is revealed the auto-action priority is `Delve` then `Forage`.
- Keep current decision: no idle stamina regen, and no extra anti-softlock guard beyond player choice.

### 6. Temple Ruins rebuild flow
- Add new hub structure def `templeRuins`.
- `templeRuins` should:
  - have title `Temple Ruins`
  - have tag `build`
  - own an inventory
  - have a pre-seeded active build process targeting `makeshiftShelter`
  - have zero material requirements
  - take `5` seconds of work
  - spend `1` stamina per contributing pawn per second via existing build passive worker cost
- Reuse existing `finalizeBuildProcess` transformation so the same instance becomes `makeshiftShelter`.
- Inventory remains because owner inventory is keyed by unchanged `instanceId`.
- Adjust hub structure hover/title logic so a pre-seeded build process can preserve the visible structure name `Temple Ruins` instead of forcing a `(Construction)` label for this case.

### 7. Naming and rename gating
- Hub header is hidden until `hubVisible`.
- While visible but before shelter completion:
  - hub header displays `Ancient Ruins`
  - clicking rename is disabled
- On `Temple Ruins -> Makeshift Shelter` completion:
  - set shared hub area name to `Hub`
  - set `hubRenameUnlocked = true`
- The board envStructure on the Levee should read from the same hub area name source, so it automatically changes:
  - `Ancient Ruins` before shelter completion
  - `Hub` after completion
  - any later user rename
- Assumption: the hub header and Levee envStructure title sync; the hub structure card itself keeps its structure-def title (`Temple Ruins`, then `Makeshift Shelter`).

### 8. View changes
- Board view filters env tiles, env events, env structures, and env-row pawns by exposed columns.
- Visible env width is derived from highest exposed frontier column, so true board size stays concealed.
- Unrevealed visible tiles render:
  - title `???`
  - generic hidden styling
  - only visible local tags such as `Explore`
- Exposed envStructures on unrevealed tiles still render normally.
- Hub chrome and hub structures render only when `hubVisible`.
- Hub header clickability/cursor respects `hubRenameUnlocked`.

## Test Cases and Scenarios
- Scenario init snapshot:
  - pawn starts at env col `0`
  - only env col `0` is exposed
  - hub row hidden
  - `Temple Ruins` exists in hidden hub row inventory with tome + mote
  - pawn inventory is empty of those two items
- Explore completion:
  - exactly 5 simulated seconds of occupied work
  - costs 1 stamina per second
  - tile title flips from `???` to real def name at completion second
  - sample package contents are deterministic across replay
  - adjacent hidden env col becomes exposed frontier
- Levee special case:
  - exposing env col `1` reveals the `Ancient Ruins` envStructure immediately
  - exploring env col `1` reveals `Delve`
  - `Delve` is ordered ahead of `forageable`
- Hub reveal:
  - `Delve` completion makes hub row/header/Temple Ruins visible
  - hub header shows `Ancient Ruins`
- Temple rebuild:
  - occupying `Temple Ruins` advances existing build process
  - 5 seconds of work transforms to `Makeshift Shelter`
  - inventory contents survive unchanged
  - `canRest` becomes available after completion
- Naming:
  - rename click disabled before shelter completion
  - completion auto-sets header/envStructure name to `Hub`
  - later rename updates both hub header and Levee envStructure title
- Replay / serialization:
  - serialize after each stage, deserialize, rebuild to same `tSec`, and confirm identical discovery state, titles, tags, inventories, and process progress
- Verification:
  - run `npm run verify`
  - add/adjust deterministic model tests for completion effects and scenario snapshot setup

## Assumptions and Defaults Chosen
- `Delve` is a tile tag on the Levee column, not an envStructure tag.
- Sample rewards use hybrid behavior: default `15` forage-table rolls, with optional future overrides.
- Unrevealed env columns are physically hidden from the rendered board; the board expands rightward as frontier is exposed.
- Hub row is fully hidden until `Delve`.
- No idle stamina regeneration is added.
- No extra pre-hub softlock protection is added beyond the chosen pacing/order.
- `Explore`, `Delve`, and the Temple rebuild all use `1` stamina per second.
- `Temple Ruins` rebuild duration is `5` seconds.
- Shared rename sync applies to hub header + Levee ruins marker, not to the hub structure card title itself.
